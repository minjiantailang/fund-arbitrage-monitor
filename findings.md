# 基金套利监控应用 - 研究发现

## 项目背景

### 需求分析
- 用户需要监控ETF和LOF基金的套利机会
- 主要关注净值与市场价格差异
- 需要手动数据更新功能
- 界面要求：仪表盘 + 基金列表筛选器

### 技术选择理由
1. **Python + PyQt**：金融数据处理能力强，跨平台支持好
2. **东方财富API**：免费、稳定、数据全面
3. **SQLite**：轻量级，适合桌面应用本地存储
4. **MVC架构**：分离关注点，便于维护和测试

- `QMainWindow`：主窗口框架
- `QTableView`：基金列表显示
- `QChart`：数据可视化
- `QThread`：后台数据处理

### 套利计算逻辑
**ETF套利公式**：
```
价差% = (市场价格 - 基金净值) / 基金净值 × 100%
潜在收益率 = 价差% - 交易费用%
```

**LOF套利公式**：
```
场内场外价差% = (场内价格 - 场外净值) / 场外净值 × 100%
考虑申购赎回费用后的净收益
```

**交易费用估算**：
- ETF：佣金0.03% + 印花税0.1% = 约0.13%
- LOF：申购费1.5% + 赎回费0.5% = 约2.0%

## 架构设计发现

### 数据流设计
```
用户操作 → 控制器 → 数据获取 → 套利计算 → UI更新
    ↓          ↓          ↓          ↓         ↓
刷新按钮 → MainController → DataFetcher → Calculator → Dashboard
```

### 数据库设计要点
**表结构**：
1. `funds`表：基金基本信息（代码、名称、类型、成立日期等）
2. `fund_prices`表：历史价格数据（时间、净值、价格）
3. `arbitrage_opportunities`表：套利机会记录
4. `user_settings`表：用户配置

**索引优化**：
- `funds.code`：主键索引
- `fund_prices.fund_code + timestamp`：复合索引
- `arbitrage_opportunities.spread_pct`：排序索引

### UI设计发现
**响应式布局策略**：
- 使用`QHBoxLayout`和`QVBoxLayout`组合
- `QSizePolicy`控制组件伸缩行为
- `QScrollArea`处理内容溢出

**性能优化**：
- 虚拟滚动处理大量数据
- 图片和图标资源缓存
- 异步数据加载防止界面卡顿

### 技术挑战
1. **API限制**：东方财富API有频率限制
   - 解决方案：添加请求间隔，使用缓存数据
   - 备用方案：实现模拟数据生成器

3. **数据一致性**：多线程数据更新可能冲突
   - 解决方案：使用线程安全的数据结构
   - 同步机制：信号/槽确保UI线程安全

## 性能瓶颈分析发现 (2026-01-31)

### 问题背景
用户报告：集成东方财富真实数据源后，程序启动和数据刷新时间过长。

### 代码分析发现

#### 1. 数据获取器 (`src/models/data_fetcher.py`)
**主要性能问题**：
- `EastMoneyDataFetcher._ensure_fund_list()`：每次启动时加载全量基金列表
  - 调用 `self.api.get_fund_list()` 获取所有基金
  - 缓存时间1小时，但启动时仍需加载
  - 基金数量可能数千个，API响应较大

- `EastMoneyDataFetcher._ensure_nav_cache()`：加载全量净值数据
  - 调用 `self.api.get_all_fund_nav()` 获取所有基金净值
  - 缓存时间4小时，但启动时仍需加载
  - 分页获取，每页20000条，可能需要多页请求

- `fetch_etf_data()` 和 `fetch_lof_data()`：串行处理
  - 先获取基金列表，再获取净值缓存，最后获取实时行情
  - 对于大量基金，网络请求次数多

#### 2. 东方财富API (`src/data_sources/eastmoney_api.py`)
**API调用分析**：
- `get_fund_list()`：单次请求获取所有基金
  - URL: `http://fund.eastmoney.com/js/fundcode_search.js`
  - 返回所有基金（开放式+封闭式），数据量大

- `get_all_fund_nav()`：分页获取净值数据
  - URL: `http://fund.eastmoney.com/Data/Fund_JJJZ_Data.aspx`
  - 参数：`page=1,20000` 每页20000条
  - 可能需要多页请求，每页都是完整请求

- `get_realtime_quotes()`：批量获取行情
  - URL: `http://push2.eastmoney.com/api/qt/ulist.np/get`
  - 批量大小：90个基金/请求
  - 相对高效，但需要等待所有批次完成

#### 3. 基金管理器 (`src/models/fund_manager.py`)
**数据处理流程**：
- `refresh_all_data()`：串行处理ETF和LOF数据
  - 先获取所有ETF数据，再获取所有LOF数据
  - 每个基金单独处理：保存基本信息、计算套利、保存价格记录
  - 数据库操作频繁

#### 4. 数据库操作 (`src/models/database.py`)
**潜在问题**：
- 每个基金的价格记录都单独保存
- 缺少批量插入优化
- 索引可能不够优化

## 性能优化实施结果 (2026-01-31)

### 学生完成的优化工作

#### 1. 全局持久化缓存（拒绝重复加载）
**优化前**：缓存仅存在于内存中，每次重启程序都需要重新从网络下载所有数据。
**优化后**：利用 SQLite 数据库将基金列表和净值数据保存到本地硬盘。程序重启后会优先读取本地数据库，实现**"秒级启动"**。

#### 2. 智能懒加载与按需获取（只取所需的）
**优化前**：每次更新净值时，会下载并解析全市场 20,000 多只基金的数据，哪怕只需监控 2,000 多只场内基金。
**优化后**：修改了 API 逻辑，仅针对筛选出的场内基金（ETF 和 LOF，约 2,349 只）进行精确的数据抓取和解析。这使得数据处理量减少了约 90%。

#### 3. 缓存策略升级（更长效的保存）
- **基金列表**：缓存有效期延长至 24 小时（原 1 小时）
- **净值数据**：缓存有效期延长至 12 小时（原 4 小时）

   - 获取5个ETF基金：需要基金列表 + 净值数据 + 实时行情

- **每次启动加载**：26,091基金信息 + 22,081净值数据 + 实时行情

### 优化方案设计

基于实测性能数据，提出以下优化方案：

#### 1. 缓存策略优化（最高优先级）
**问题**：每次创建`EastMoneyDataFetcher`都重新加载全量数据
**解决方案**：
- **全局缓存共享**：使用单例模式或模块级缓存，避免重复加载
- **延长缓存时间**：
  - 基金列表：24小时（数据变化慢）
  - 净值数据：12小时（每日更新）
  - 实时行情：5分钟（高频变化）
- **磁盘缓存**：将缓存数据保存到文件，避免重启后重新加载
- **增量更新**：只更新变化的数据，避免全量刷新

#### 2. 懒加载策略优化
**问题**：启动时加载26,091个基金信息，但实际只关注1,720个ETF和629个LOF
**解决方案**：
- **按需加载**：启动时只加载场内基金（ETF+LOF，约2,349个）
- **后台加载**：其他基金数据在后台线程中加载
- **分类缓存**：分开缓存场内基金和场外基金数据

#### 3. 并发处理优化
**问题**：串行处理导致ETF数据获取异常缓慢（2.8秒/基金）
**解决方案**：
- **并行API调用**：使用`concurrent.futures.ThreadPoolExecutor`并发获取数据
- **批量处理**：将基金分组批量获取，减少网络往返
- **异步UI更新**：数据获取不影响UI响应

#### 4. 请求优化
**问题**：获取5个ETF基金需要多个独立请求
**解决方案**：
- **请求合并**：将基金列表、净值数据、实时行情请求合并
- **减少字段**：只请求必要字段，减少响应大小
- **智能重试**：优化重试策略，避免不必要的等待
- **连接复用**：保持HTTP连接，减少握手开销

#### 5. 架构优化
**问题**：`DataFetcher`实例创建成本高
**解决方案**：
- **单例模式**：确保只有一个`EastMoneyDataFetcher`实例
- **依赖注入**：通过工厂模式管理实例生命周期
- **资源池**：重用网络连接和解析器

#### 6. 数据库优化
**问题**：频繁的数据库操作
**解决方案**：
- **批量插入**：使用SQLite的批量插入功能
- **事务处理**：将多个操作放在一个事务中
- **索引优化**：确保常用查询有合适索引
- **数据清理**：定期清理历史数据，保持数据库精简
