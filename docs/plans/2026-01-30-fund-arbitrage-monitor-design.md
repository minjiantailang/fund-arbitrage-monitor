# 基金套利监控桌面应用 - 设计文档

## 项目概述

**目标**：开发一个基于Python的桌面应用，实时监控ETF和LOF基金的套利机会，帮助投资者发现并评估净值与市场价格之间的套利机会。

**核心功能**：
1. 监控ETF基金净值与市场价格差异
2. 监控LOF基金场内交易价格与场外申购赎回净值差异
3. 计算套利机会和潜在收益
4. 提供实时监控仪表盘和基金列表筛选器

**技术栈**：Python + PyQt/PySide + Pandas + SQLite

---

## 架构设计

### 1. 系统架构（MVC模式）

#### 模型层（Model）
- **DataFetcher类**：封装东方财富/雪球API调用，获取基金净值、价格数据
- **ArbitrageCalculator类**：计算套利机会、价差、潜在收益
  - ETF套利公式：`价差% = (价格 - 净值) / 净值 × 100%`
  - LOF套利公式：`场内场外价差% = (场内价格 - 场外净值) / 场外净值 × 100%`
- **FundManager类**：管理基金列表、筛选条件、数据缓存

#### 视图层（View）
- **DashboardWidget**：实时监控仪表盘，显示汇总统计和关键指标
- **FundListWidget**：基金列表与筛选器，支持搜索、排序、筛选操作

#### 控制器层（Controller）
- **MainController**：处理用户操作（刷新数据、筛选基金等）
- **DataRefreshController**：管理手动数据更新流程

### 2. 数据流程

1. **手动触发更新**：用户点击"刷新"按钮 → 控制器调用`DataFetcher`
2. **API数据处理**：从东方财富API获取ETF净值/价格、LOF场内场外数据
3. **套利计算**：`ArbitrageCalculator`计算价差和收益率
4. **UI更新**：通过信号/槽机制更新仪表盘和基金列表

### 3. 核心功能模块

- **基金筛选器**：按基金类型（ETF/LOF）、代码、名称、价差范围筛选
- **排序功能**：按价差%、收益率、基金规模等字段排序
- **状态指示器**：颜色编码表示套利机会（绿：机会，黄：观察，红：无机会）
- **数据缓存**：本地SQLite数据库缓存历史数据，支持离线查看

### 4. 异常处理

- API调用失败时降级到缓存数据
- 网络超时自动重试机制
- 数据格式验证和错误日志记录

## UI界面设计

### 1. 主界面布局

**左侧栏（70%宽度） - 仪表盘区域**：
- 顶部统计卡片：显示"发现套利机会数"、"最高价差%"、"平均收益率"
- 中间价差分布图表：柱状图展示不同价差区间的基金数量
- 底部关键基金列表：前5个最佳套利机会（代码、名称、价差%、收益率）

**右侧栏（30%宽度） - 基金列表区域**：
- 顶部搜索栏和筛选控件（基金类型选择、价差范围滑块）
- 中间表格展示所有基金数据（代码、名称、类型、最新净值、价格、价差%、收益率）
- 底部操作栏："刷新数据"、"导出CSV"、"设置"按钮

### 2. 交互设计

- **双击基金行**：显示该基金的详细历史图表（新窗口）
- **右键菜单**："加入观察列表"、"查看详情"、"复制代码"
- **实时状态指示**：表格行背景色根据价差%渐变（浅绿→深绿→黄色→红色）
- **筛选联动**：仪表盘统计实时更新基于当前筛选条件

### 3. 视觉设计

- 金融应用配色方案（深色/浅色主题可选）
- 清晰的数字字体和适当的间距
- 响应式布局，支持窗口缩放

## 技术实现细节

### 1. 数据源接口

**东方财富API集成**：
- 基金净值查询接口
- 实时行情数据接口
- LOF场内外数据接口

**数据格式**：
```python
{
    "fund_code": "510300",
    "fund_name": "沪深300ETF",
    "fund_type": "ETF",
    "nav": 3.56,          # 单位净值
    "price": 3.58,        # 市场价格
    "spread_pct": 0.56,   # 价差百分比
    "yield_pct": 0.45,    # 收益率（扣除费用）
    "update_time": "2026-01-30 14:30:00"
}
```

### 2. 数据库设计

**SQLite表结构**：
- `funds`：基金基本信息表
- `fund_prices`：历史价格数据表
- `arbitrage_records`：套利机会记录表
- `user_settings`：用户配置表

### 3. 性能考虑

- 数据更新采用异步处理，避免界面卡顿
- 数据库查询使用索引优化
- 内存缓存频繁访问的数据
- 批量处理大量基金数据更新
