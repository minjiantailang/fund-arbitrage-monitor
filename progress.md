## 2026-01-31 性能瓶颈分析

### 问题报告
用户反馈：集成东方财富真实数据源后，程序启动和数据刷新时间过长。

### 分析过程
1. **代码审查**：检查了关键文件：
   - `src/models/data_fetcher.py` - 数据获取器实现
   - `src/data_sources/eastmoney_api.py` - 东方财富API封装
   - `src/models/fund_manager.py` - 基金管理器
   - `src/models/database.py` - 数据库操作

2. **性能瓶颈识别**：
   - **启动时的全量数据加载**：基金列表 + 全量净值数据 + 实时行情
   - **串行处理流程**：ETF和LOF数据串行获取，缺乏并发
   - **网络请求过多**：多个独立API调用，缺乏合并
   - **数据库操作频繁**：每个基金单独处理，缺少批量操作

3. **数据量估计**：
   - 基金总数：约8000+（开放式基金）
   - ETF数量：约500+（51/56/58/159开头）
   - LOF数量：约300+（16/50开头）
   - 每次启动需要加载：8000+基金信息 + 8000+净值数据 + 800+实时行情

### 优化建议
1. **缓存策略优化**：
   - 延长缓存时间（基金列表24小时，净值数据12小时）
   - 实现增量更新，避免全量刷新
   - 使用更高效的缓存存储格式

2. **并发处理优化**：
   - 使用 `concurrent.futures` 或 `asyncio` 并发获取数据
   - ETF和LOF数据并行获取
   - 批量处理数据库操作

3. **懒加载策略**：
   - 启动时只加载必要数据（如活跃基金）
   - 按需加载详细数据
   - 后台预加载其他数据

4. **请求优化**：
   - 合并相关API调用
   - 减少不必要的字段请求
   - 优化重试策略和超时设置

5. **数据库优化**：
   - 批量插入操作
   - 优化索引策略
   - 定期清理历史数据
